# Design: Data Engineering Domain Profile

## Technical Approach

The domain profile is a **configuration + branching mechanism layered on top of the untouched SDD 8-phase core**, not a fork. It follows gentle-ai's established **authoritative-state pattern**: Go code produces machine-readable state (`gentle-ai sdd-status --json` is already "authoritative over prompt inference"), and skills branch on that state rather than inferring from prompts. The domain profile extends this pattern with a new `gentle-ai sdd-config --json` CLI backed by a pure-function Go core, so every branch point is unit-testable under strict TDD (`go test ./...`).

The 4 app-dev layers (skills, verify, spec templates, tools) gain a `domain: data-engineering` branch. When `domain` is unset, every code path is identical to today — backward-compatible by construction. The 5 PRs are vertical slices, each landing independently behind the config gate.

## Architecture Decisions

| Decision | Choice | Rejected | Rationale |
|---|---|---|---|
| Domain gate location | New Go core `internal/sddconfig/` + `internal/etl/` exposed via `gentle-ai sdd-config --json`; skills branch on JSON | Prompt-only branching in skills | Strict TDD requires Go-testable branch points; `openspec/config.yaml` is currently unparsed by Go — a pure-function reader + CLI mirrors the proven `sdd-status` bridge and kills "magic" inference |
| 7 data-engineer skills delivery | **Embed** all 7 into `internal/assets/skills/` (copy + patch), register in `internal/catalog/skills.go`; new `data-engineer-pattern-detect` embedded too | Patch user-only copies in `~/.config/opencode/skills/` (not PR-able) | Proposal lists them as Modified + delivered via PR; embedding makes them versioned/installable/PR-able; fixes the live authorship violation (`data-engineer-etl-s3` ships `# author: generated by gentle-ai`) |
| Shared protocol block | `internal/assets/skills/_shared/data-engineer-protocol.md` referenced by each skill | Inline duplication in 7 skills | DRY; keeps per-skill Phase-4 delta small enough for the 800-line budget |
| Verify branching | Mode-branch inside `sdd-apply` (Camino A) and `sdd-verify` (Camino B), gated on `sdd-config` domain | Separate `data-engineer-verify` skill | Matches user-confirmed Q3; one entry point; Go core owns the gate so the skill stays thin |
| Spec format | OpenSpec delta (behavior) + `glue-tables/{db}.{table}.yaml` sidecar (schema) | Pure delta / pure YAML | User-confirmed Q4; sidecar validated structurally by Go `ValidateSidecar` against `aws glue get-table` output |
| Pattern detection | New `data-engineer-pattern-detect` skill backed by Go `internal/etl/pattern.go` heuristics | Logic inside `data-engineer-integrate` | User-confirmed Q6; pure-function heuristics are table-testable; confidence + override never silent |

## Data Flow

```
 openspec/config.yaml  ──┐
  domain/repos/aws_profiles
                         │  (parsed by internal/sddconfig)
                         ▼
          gentle-ai sdd-config --json  ◄── authoritative domain state
                         │
   ┌─────────────────────┼─────────────────────┐
   ▼                     ▼                     ▼
 sdd-init           sdd-spec/design        sdd-apply / sdd-verify
 (detect+confirm)   (ETL sections+         (Camino A: Glue Docker TDD
                    sidecar template)       Camino B: SAM+Athena EXCEPT)
                         │                     │
                         ▼                     ▼
              internal/etl/* pure funcs   internal/etl/* pure funcs
              (pattern, sidecar,          (header, sidecar-validate,
               prefix, gitflow)            compare-SQL, scrub, scaffold)
```

The orchestrator, `sdd-status`, and the 8-phase core are NOT in this flow — they are untouched. Skills call `gentle-ai sdd-config --json` (or read parsed struct via the skill's bash/read) and follow a `domain == data-engineering ? caminoB : appdev` decision gate.

## Interfaces / Contracts

```go
// internal/sddconfig
type Config struct {
    Domain      string              // "data-engineering" | "" (app-dev)
    Repos       Repos               // {Infra, Carga string}
    AWSProfiles map[string]string   // logical "dev" -> CLI profile name
    Verify      VerifyOpts          // {SkipDeploy bool}
}
func LoadConfig(root string) (Config, error)          // missing fields = zero (app-dev)
func DetectDomain(root string) (domain string, confidence float64, evidence []string, err error)

// internal/etl
type ETLHeader struct {
    JobName, DesarrolladoPor, FechaCreacion,
    ModificadoPor, FechaModificacion, Descripcion string
}
func ValidateHeader(content string) error             // rejects "generated by", "Co-Authored-By", AI names
func UpdateHeaderForModify(orig, human, today string) string // preserves desarrollado_por+fecha_creacion

func DetectPattern(markers Markers) (Pattern, float64, string)

func RepoPrefix(target string) string   // "infra" | "carga" | "both"
func BuildExceptSQL(target, devDB, prdDB string) string
func ScrubProfiles(text string, cfg sddconfig.Config) string
```

## Per-Phase Implementation (HOW)

**Phase 1 — Detection + config (`sddconfig` core + `sdd-init`).** `DetectDomain` scans root for `template.yaml` AND `glue-jobs/*.py` (both → `data-engineering`, confidence ~0.8; one → ~0.5 hint). `sdd-init` skill gains a preflight gate: run `gentle-ai sdd-config --detect`, present hint, confirm/override, write `domain` to config. `LoadConfig` is the single reader; `ValidateRepos` warns on missing paths each session. No behavior change until phases 2-5 read `domain`. CLI: `gentle-ai sdd-config --json`.

**Phase 2 — Spec + design templates (`sdd-spec`, `sdd-design`).** When `domain=data-engineering`, `sdd-spec` emits the 6 ETL delta sections (`## Source Tables`, `## Target Schema`, `## Watermark Strategy`, `## DAG`, `## AWS Profile Requirements`, `## Verify Approach`) + one sidecar per target table. Behavioral scenarios assert table output (data comparison), not function output. `sdd-design` emits a DAG-of-transformations (nodes = sources/temp-views/target, edges = deps) and, for modifications, marks the insertion point + affected downstream stages. Templates live in the skill bodies; the Go `etl` package validates generated artifacts.

**Phase 3 — Apply + Verify dual-path (`sdd-apply`, `sdd-verify`).** Both read `sdd-config` domain and branch. Camino A (`sdd-apply`): Glue Docker `aws-glue-libs:5`, throwaway `_test` table, TDD loop whose scaffold is selected by `DetectPattern` (watermark test / multi-step DAG test / Athena-Spark parity test). Every generated/modified `.py` passes `ValidateHeader`; `UpdateHeaderForModify` preserves authorship. Camino B (`sdd-verify`): `sam deploy` both repos (parallel; `verify.skip_deploy` short-circuits to sidecar+SQL-only), run job, `BuildExceptSQL` dev-vs-prd, `ValidateSidecar` against `aws glue get-table`. Report output runs through `ScrubProfiles`. Branch is a single `if domain == "data-engineering"` gate per skill — app-dev path untouched.

**Phase 4 — Skills update (embed 7 + new + protocol block).** Copy the 7 `data-engineer-*` into `internal/assets/skills/`, prepend `## Reference: _shared/data-engineer-protocol.md`, and fix the authorship violation (`etl-s3` `# author: generated by gentle-ai` → human `desarrollado por`). New `data-engineer-pattern-detect` skill wraps `etl.DetectPattern`, reports confidence, offers 4-pattern menu. Register all 8 in `model` SkillIDs + `catalog/skills.go`. **Risk**: 8 skills may exceed 800 lines → split into 4a (embed+protocol+pattern-detect+catalog) / 4b (patch 7 bodies); `sdd-tasks` decides.

**Phase 5 — Tasks multi-repo + git flow (`sdd-tasks`).** `RepoPrefix(taskTarget)` emits `[infra]`/`[carga]`/`[both]` per task (target touches `glue-tables/`→infra, `glue-jobs/`→carga, both→both). `GitFlowForRepo` annotates: master-project tasks → GitHub flow (`branch-pr` applies); company-repo tasks → Bitbucket `feature/ → develop → release/` (`branch-pr` does NOT apply). `sdd-tasks` reads `repos:` via `sddconfig.LoadConfig` and validates paths exist (drift detection). Depends on Phase 1 config only.

## Testing Strategy

| Layer | What | Approach |
|---|---|---|
| Unit | `sddconfig` parse/detect/validate/scrub; `etl` header/sidecar/pattern/compare/prefix/gitflow | Table-driven `*_test.go`; temp dirs for filesystem scans; golden files for header/sidecar render |
| Unit | Backward-compat | `LoadConfig` on app-dev `config.yaml` (no `domain`) returns zero values; all branch funcs return app-dev defaults |
| Unit | Authorship enforcement | `ValidateHeader` rejects `Co-Authored-By`, `generated by`, AI model names; `UpdateHeaderForModify` preserves original author |
| Integration | `sdd-config --json` CLI | `cli` package test pattern (mirrors `sdd_status_test.go`) |
| Integration | Both verify branches | Shared fixtures in skill `testdata/` exercising app-dev AND data-eng branch from one config family |
| E2E | Camino A/B | Out of Go `go test` scope (needs Docker/AWS); covered by skill orchestration + `verify.skip_deploy` local mode |

Strict TDD: every Go function above is written test-first. Skills (prompts) are validated by the spec scenarios in `sdd-verify`, not by `go test`.

## Risk Resolution (8 exploration risks)

| Risk | Mitigation (Go-testable unless noted) |
|---|---|
| Bifurcation drift | Single gate source `sddconfig.Domain`; shared fixtures run both branches; CI runs both |
| Pattern misclassification | `DetectPattern` returns confidence; user override in `sdd-propose`; never silent; table tests for all 4 + ambiguous |
| Sidecar ↔ delta drift | `ValidateSidecar` vs `aws glue get-table`; delta names sidecar file; structural mismatch report |
| Repo path fragility | `ValidateRepos` each session; `sdd-init` warns; temp-dir unit tests |
| SAM deploy cost | Parallel deploys (skill orchestration); `verify.skip_deploy` flag; cached stack outputs |
| Bitbucket vs GitHub PR | `GitFlowForRepo` per prefix; `branch-pr` scoped to master only; annotated in `tasks.md` |
| Authorship enforcement | `RenderHeader`/`ValidateHeader` from session config; rejects AI attribution; fixes live `etl-s3` violation |
| AWS profile leak | `ScrubProfiles` on all reports/logs; skills use logical names only |

## Migration / Rollback

No migration required. All config fields are optional; absent `domain` = app-dev = today. Per-phase revert = `git revert <PR>`; phases ordered, mostly independent (Phase 5 depends on Phase 1 only). Embedded skills are additive; deleting the new branch restores prior behavior. `data-engineer-pattern-detect` and the 7 newly-embedded skills can be removed with no downstream dependency.
