# Data Engineer Protocol — Shared Reference

> Shared contract referenced by every `data-engineer-*` skill. Each skill
> prepends `## Reference: _shared/data-engineer-protocol.md` to point readers
> here. Keep this file DRY: change the rule once, every skill inherits it.

## 1. ETL Header Protocol

Every generated or modified ETL `.py` file in the CARGA repo starts with a
6-field comment header rendered by `internal/etl.RenderHeader` and validated by
`internal/etl.ValidateHeader`. The header is the single source of truth for
job identity and authorship.

Format (label-column aligned, rune-aware padding so accented vowels line up):

```python
# nombre del job       : <job-name>
# desarrollado por     : <human-author>
# fecha creación       : <YYYY-MM-DD>
# modificado por       : <human-modifier-or-same-as-author>
# fecha modificación   : <YYYY-MM-DD>
# descripción          : <one-line purpose>
```

- Field order is fixed; column alignment is computed by `RenderHeader`.
- `ValidateHeader` rejects: `co-authored-by`, `generated by`,
  `auto-generated`, and explicit AI vendor/model names (`claude`, `gpt`,
  `chatgpt`, `gemini`, `copilot`, `openai`, `anthropic`). The forbidden
  vocabulary is deliberately NOT a bare `ai` substring (it would false-fire
  on common ETL words like `available`, `domain`, `certain`).
- `UpdateHeaderForModify(orig, human, today)` preserves
  `desarrollado por` + `fecha creación` and updates only
  `modificado por` + `fecha modificación`. On unparseable input it returns
  `orig` unchanged (non-destructive fallback).

## 2. Authorship Rule — AI is a Tool, Never the Author

Generated ETL code is ALWAYS authored under a human name. The agent may write
the bytes, but the header never attributes the work to an AI. Specifically:

- ❌ Forbidden: `# author: generated by gentle-ai`
- ❌ Forbidden: `# desarrollado por: Claude` / `GPT` / `Copilot` / any model
- ❌ Forbidden: `# co-authored-by: dependabot` / any bot
- ✅ Required: `# desarrollado por: <human-name>` (the engineer directing the
  agent for that change)

When the existing source already carries a violation (e.g. the legacy
`data-engineer-etl-s3` shipped `# author: generated by gentle-ai`), the
embedding skill MUST replace it with a human-name placeholder
(`<human-author>`) so the violation is not propagated into the project.

## 3. Pattern Index — 4-Step ETL Taxonomy

`internal/etl.DetectPattern(markers) (Pattern, float64, rationale)` classifies
an ETL job into exactly one of four patterns (or `unknown` with rationale).
The taxonomy drives scaffold selection in `sdd-apply` Camino A.

| Pattern            | Markers                                            | Confidence | Scaffold                |
|--------------------|----------------------------------------------------|------------|-------------------------|
| `incremental`      | `from_iso`/`created_dynamic_frame.from_catalog` + watermark column | 0.8        | Watermark test          |
| `multi-step`       | `GlueContext` + temp views (`createOrReplaceTempView`) + `!awswrangler.athena` | 0.8        | Multi-step DAG test     |
| `legacy-wrangler`  | `awswrangler.athena.read_sql` + Athena predicate pushdown | 0.8        | Legacy-wrangler migrate |
| `glue-studio`      | `ApplyMapping` + `SparkSqlQueryHelper` (both required) | 0.8        | Athena-Spark parity     |

- Confidence is 0.8 when both markers fire, 0.5 when only one fires (a hint,
  not a verdict), 0 when no markers fire.
- When two patterns both match, the higher-confidence wins; ties fall back to
  `unknown` with a rationale naming the partial signals. Detection is NEVER
  silent — every result carries a human-readable rationale.
- The user may override the detected pattern during `sdd-propose`. Overrides
  are recorded in the proposal and never silently dropped.

## 4. Master-Project Awareness — Where Each Skill Runs

The Toyota Chile data platform splits work across two repos, both resolved
through `internal/sddconfig.LoadConfig`:

| Repo   | `repos.<key>` | Contains                          | Git flow             |
|--------|---------------|-----------------------------------|----------------------|
| master | `repos.infra` | `template.yaml`, `glue-tables/*.yaml` | GitHub flow (`branch-pr` applies) |
| carga  | `repos.carga` | `glue-jobs/*.py`, SAM stack for ETL   | Bitbucket flow (`feature/ → develop → release/`; `branch-pr` does NOT apply) |

- Every data-engineer skill declares which repo it runs against at the top of
  its workflow. Skills that touch BOTH repos (e.g. `data-engineer-create-table`
  three-phase workflow) sequence the phases explicitly.
- `RepoPrefix(taskTarget)` in `internal/etl/tasks.go` (Phase 5) emits the
  `[infra]` / `[carga]` / `[both]` task prefix; until then, skills annotate
  phases inline.
- `GitFlowForRepo(repo)` decides whether `branch-pr` is available: master →
  yes, carga → no (Bitbucket flow instead).

## 5. AWS Profile — Logical-Name Rule

ETL code and skill prose NEVER hardcode AWS CLI profile names or 12-digit
account IDs. The mapping lives in `openspec/config.yaml` under `aws_profiles`
and is resolved at runtime by `internal/sddconfig.ResolveProfile(cfg, logical)`:

```yaml
aws_profiles:
  prd:     AWSReadFullDat-874970050509   # production read access
  dev:     aws-tcl-ope-set-cloud-895593169121   # dev stage
  usuario: <usuario-profile>             # per-engineer scratch profile
```

- In skill prose and generated ETL, ALWAYS reference the **logical** name:
  `ResolveProfile(cfg, "dev")`, never the concrete profile string.
- `ScrubProfiles(text, cfg)` strips both the concrete profile names AND bare
  12-digit account IDs from any report, log, or verify output before it
  leaves the pipeline. Apply it on every artifact that may be shared
  externally (PR descriptions, verify reports, issue comments).
- The three logical names are a closed set: `prd`, `dev`, `usuario`. New
  logical names require extending this file AND `ResolveProfile`.
