## When to Use

- When source system provides only business rules documentation, not SQL
- When transformation logic needs to be formalized before ETL coding
- When existing SQL needs to be refactored into readable CTEs
- When ETL transformation is complex and needs multi-step documentation
- During `sdd-explore` or early `sdd-spec` phases when transformation logic needs formalization

## Critical Patterns

### CTE Naming Convention
Use logical, descriptive names for each transformation step:
- `filtered_data` — WHERE conditions applied
- `enriched_with_*` — JOINs or calculations added
- `deduplicated` — duplicates removed
- `aggregated_*` — GROUP BY and aggregates applied
- `windowed_*` — window functions applied
- `final_output` — final SELECT for ETL output

### SparkQL Compatibility
**Glue uses Spark SQL, NOT Presto or PostgreSQL.**
- Use `CAST(col AS INT)` not `col::int`
- Use `date_format()` not `TO_CHAR()`
- Use `NVL()` or `COALESCE()` for null handling
- Window functions: `ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...)` is standard

### File Output Location
Save to `artefactos/<table_name>.sql` — **outside official CARGA/INFRA repos**
- `artefactos/` is the staging area for generated SQL
- Not committed to version control in official repos
- Reference in ETL code via absolute path

### Header Comment Block
Every SQL file MUST have this header:

```sql
-- ============================================================================
-- File: artefactos/<table_name>.sql
-- Description: {one-line description of transformation}
-- Source: {business logic source}
-- Author: generated by gentle-ai
-- Date: {YYYY-MM-DD}
-- ============================================================================
```

## Code Examples

### Basic CTE Structure

```sql
WITH filtered_data AS (
    SELECT *
    FROM source_table
    WHERE status = 'ACTIVE'
      AND date >= '2024-01-01'
),
enriched_with_period AS (
    SELECT
        *,
        CASE
            WHEN amount > 1000 THEN 'high'
            WHEN amount > 500 THEN 'medium'
            ELSE 'low'
        END AS amount_category
    FROM filtered_data
),
final_output AS (
    SELECT
        id,
        name,
        date,
        amount_category,
        CAST(amount AS DECIMAL(18,2)) AS amount
    FROM enriched_with_period
)
SELECT * FROM final_output
```

### Aggregation with GROUP BY

```sql
WITH daily_sales AS (
    SELECT
        DATE_TRUNC('day', sale_date) AS sale_day,
        product_category,
        SUM(quantity) AS total_qty,
        SUM(amount) AS total_amount,
        COUNT(DISTINCT customer_id) AS unique_customers
    FROM source_sales
    GROUP BY DATE_TRUNC('day', sale_date), product_category
),
final_output AS (
    SELECT
        sale_day,
        product_category,
        total_qty,
        total_amount,
        unique_customers,
        ROUND(total_amount / NULLIF(unique_customers, 0), 2) AS avg_order_value
    FROM daily_sales
)
SELECT * FROM final_output
```

### Window Functions

```sql
WITH ranked_transactions AS (
    SELECT
        customer_id,
        transaction_date,
        amount,
        ROW_NUMBER() OVER (
            PARTITION BY customer_id
            ORDER BY transaction_date DESC
        ) AS rn,
        SUM(amount) OVER (
            PARTITION BY customer_id
            ORDER BY transaction_date
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) AS running_total
    FROM transactions
),
final_output AS (
    SELECT
        customer_id,
        transaction_date,
        amount,
        running_total,
        -- Flag first transaction per customer
        CASE WHEN rn = 1 THEN 'new' ELSE 'returning' END AS customer_status
    FROM ranked_transactions
)
SELECT * FROM final_output
```

### Multi-Table Join with CTEs

```sql
WITH base_orders AS (
    SELECT
        o.order_id,
        o.customer_id,
        o.order_date,
        o.status
    FROM orders o
    WHERE o.order_date >= '2024-01-01'
),
customer_info AS (
    SELECT
        customer_id,
        customer_name,
        region
    FROM customers
),
order_items AS (
    SELECT
        order_id,
        SUM(quantity) AS total_items,
        SUM(amount) AS total_amount
    FROM order_items
    GROUP BY order_id
),
final_output AS (
    SELECT
        o.order_id,
        o.order_date,
        c.customer_name,
        c.region,
        i.total_items,
        i.total_amount,
        CASE
            WHEN i.total_amount > 1000 THEN 'premium'
            ELSE 'standard'
        END AS customer_tier
    FROM base_orders o
    JOIN customer_info c ON o.customer_id = c.customer_id
    JOIN order_items i ON o.order_id = i.order_id
)
SELECT * FROM final_output
```

### Date Handling (Common in Toyota Chile ETL)

```sql
WITH transformed_dates AS (
    SELECT
        id,
        -- Parse DD/MM/YYYY (Spanish locale)
        TO_DATE(date_str, 'DD/MM/YYYY') AS parsed_date,
        -- Extract year/month/day for partitioning
        YEAR(date_str, 'DD/MM/YYYY') AS year,
        MONTH(date_str, 'DD/MM/YYYY') AS month,
        DAY(date_str, 'DD/MM/YYYY') AS day,
        -- Handle null dates
        COALESCE(original_date, '1900-01-01') AS effective_date
    FROM source_table
),
final_output AS (
    SELECT
        id,
        parsed_date,
        year,
        month,
        day,
        effective_date
    FROM transformed_dates
)
SELECT * FROM final_output
```

## Commands

### Generate SQL from Business Logic

```bash
# Run sql-from-logic skill to generate SQL file
# Input: business logic description (from analyst, docs, or existing code)
# Output: artefactos/<table_name>.sql

python -c "
from pathlib import Path
# Placeholder for SQL generation from structured input
print('SQL generation requires business logic input')
"
```

### Validate Generated SQL

```bash
# Validate SparkSQL syntax using Spark (if available)
spark-sql -f artefactos/table_name.sql

# Or use Athena simulation
aws athena start-query-execution --query-string "SELECT * FROM (YOUR_SQL) LIMIT 1"
```

## Output

The skill produces a SQL file at `artefactos/<table_name>.sql` with:
- **Documentation header**: table name, author, date, description, source
- **CTEs for each transformation step**: logically named intermediate steps
- **Final SELECT**: producing `df_final` compatible output for Spark
- **SparkQL compatible syntax**: works in Glue/Athena environment
